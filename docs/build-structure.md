# Build Structure

Date: 2026-06-28

## Decisions to lock now

- Build an iOS 17+ SwiftUI app first. Do not spend MVP time on iOS 16, tvOS, macOS, CarPlay, Chromecast, cloud drives, or server-side components.
- Make the app target thin. Most logic belongs in a local Swift package so builder agents can work in separate modules and test without opening the whole app.
- Use SMB as the first real source. WebDAV is second. FTP/SFTP/NFS/DLNA wait until the media identity, queue, cache, and scan model are stable.
- Start with cache-first AVPlayer playback. Prototype range streaming before committing UI around instant remote playback. Add VLCKit only after the library, queue, cache, and offline model work.
- Use GRDB and SQLite for the library store. Do not use SwiftData/Core Data for the media index; this app needs explicit schema control, FTS, checkpoints, and predictable migrations.
- Use `SMBClient` first. Keep `AMSMB2` as a fallback spike only, because its binary and LGPL distribution story must be settled before it ships.
- Do not name a Swift module `SourceKit`; that collides conceptually with Apple tooling. The source-management module is `BetterStreamingSources`.

## Product slices

Build in product slices that each leave the app runnable:

| Slice | User outcome | Main modules |
| --- | --- | --- |
| Source setup | User can add/test an SMB source and see reachable roots. | `BetterStreamingSources`, `SMBRemote`, `RemoteFileSystem`, `Diagnostics` |
| Path library | User can scan a root and browse folders before metadata finishes. | `LibraryIndexer`, `MediaStore`, `RemoteFileSystem`, `AppUI` |
| Queue playback | User can play/shuffle a folder and keep a persistent queue. | `PlaybackCore`, `CacheManager`, `MediaStore`, `AppUI` |
| Offline cache | User can pin files/folders/playlists and play them without the source. | `CacheManager`, `PlaylistCore`, `MediaStore` |
| Search/metadata | User can search title/artist/album/path and browse albums/artists. | `LibraryIndexer`, `MediaStore`, `MetadataCore`, `AppUI` |
| Video | User can play common videos from cache/stream, with compatibility path later. | `PlaybackCore`, `StreamBridge`, optional `VLCKitBridge` |
| Diagnostics | User can understand connection speed, failures, and source health. | `Diagnostics`, `BetterStreamingSources`, `RemoteFileSystem` |

The first beta should feel like a music app with a strong folder mode. A protocol picker or generic file manager is not an acceptable first screen.

## Module boundaries

Dependency direction is one-way:

```text
App UI -> feature use cases -> service modules -> domain types
```

No service module imports SwiftUI. No storage module imports SMB. No playback module imports source setup UI.

### `BetterStreamingDomain`

Pure shared types:

- `SourceID`, `ShareID`, `RemotePath`, `RemoteItemID`
- `MediaItem`, `FolderItem`, `PlaylistID`, `QueueItem`
- `RemoteItemIdentity`
- playback enums, cache states, scan states, source health states

Keep this target small. It may import Foundation for `Date`, `URL`, and `UUID`, but it must not import GRDB, AVFoundation, Security, SwiftUI, or any network client.

### `AppFoundation`

Cross-cutting utilities:

- logging wrappers around `os.Logger`
- clock/UUID abstractions for tests
- app directory resolver
- error normalization and redaction helpers
- small async utilities

Do not put product policy here. If it knows what a playlist or SMB share is, it belongs elsewhere.

### `RemoteFileSystem`

Protocol-neutral file access:

```swift
public protocol RemoteFileSystemClient: Sendable {
    func list(_ directory: RemotePath) async throws -> [RemoteEntry]
    func stat(_ path: RemotePath) async throws -> RemoteMetadata
    func read(_ path: RemotePath, range: Range<Int64>) async throws -> Data
    func download(_ path: RemotePath, to localURL: URL, progress: ProgressSink) async throws
}
```

This target defines capabilities, entries, byte-range semantics, and fake clients for tests. It must not know about credentials, Keychain, GRDB, AVPlayer, or SwiftUI.

### `SMBRemote`

The `SMBClient` adapter:

- maps host/share/path into `RemoteFileSystemClient`
- normalizes SMB errors into app error types
- handles Unicode filenames, large directory pagination if needed, stat/read/download calls
- exposes integration-test hooks for real NAS fixtures

This module must not store credentials. It receives an authenticated connection configuration from `BetterStreamingSources`.

### `BetterStreamingSources`

Source setup and credentials:

- saved source records
- credential references
- Keychain read/write/delete
- local-network permission copy and Bonjour SMB discovery
- connection tests and source-health snapshots
- source repair entry points

Use the native Security framework directly through a small wrapper. Do not add a Keychain dependency unless the direct wrapper becomes a real maintenance problem.

### `MediaStore`

SQLite persistence through GRDB:

- schema migrations
- repositories for sources, folders, tracks, videos, playlists, queue, cache records, scan checkpoints
- FTS tables for filename/path first, then metadata
- durable queue state

This module is the only module that imports GRDB. Other modules depend on repository protocols or store APIs, not raw SQL.

### `LibraryIndexer`

Progressive indexing:

- path-first recursive traversal
- cancellation, pause/resume, checkpoints
- subtree scans
- incremental rescan and repair matching
- work scheduling for metadata extraction

It consumes `RemoteFileSystemClient` and `MediaStore`. It does not render UI and does not start playback.

### `CacheManager`

Two cache layers:

- playback byte cache for streaming and prefetch
- durable offline cache for pinned files/folders/playlists

It owns quota, eviction, partial files, stale detection, file protection, and download state. It must not choose queue order; `PlaybackCore` owns queue policy.

### `StreamBridge`

The chosen range-streaming implementation:

- either `AVAssetResourceLoader` custom scheme or a local HTTP range proxy
- content-info responses
- seek behavior
- cache-aware byte reads

Keep this separate so it can be replaced if the prototype proves the first choice wrong.

### `PlaybackCore`

Playback state and system media integration:

- `AVQueuePlayer` audio path
- player abstraction for future video/VLCKit backend
- queue, shuffle, repeat, play next, append, reorder, clear
- Now Playing metadata
- remote commands
- audio session, interruptions, route changes
- queue persistence coordination

It asks a resolver for playable assets. It should not know whether an item came from SMB, WebDAV, cache, or a stream bridge.

### `PlaylistCore`

Playlists and live folders:

- app playlists with item references
- live recursive folder playlists
- M3U import/export
- offline pin intent for playlists

Do not mix playlist editing into `PlaybackCore`; a queue is transient playback state, while a playlist is durable library state.

### `MetadataCore`

Metadata extraction and artwork:

- extension/type detection
- duration when cheap
- audio tags
- embedded artwork
- folder artwork lookup
- artwork thumbnail cache

Run this behind the path-first scan. Never block folder playback on metadata.

### `Diagnostics`

Debuggable source and playback health:

- speed tests
- server capability snapshots
- failed-read classification
- redacted debug bundle export
- credential/path redaction tests

This module can depend on source and filesystem protocols, but it must treat secrets as toxic by default.

## Data identity rules

Every persisted remote item needs identity stronger than a URL string:

```text
source_id + share_id + normalized_remote_path + remote_file_id_if_available + size + modified_at
```

Rules:

- `source_id` is stable across credential changes.
- `share_id` is stable across host alias repair when the user confirms the repair.
- `normalized_remote_path` preserves display names but compares with protocol-specific normalization.
- `remote_file_id_if_available` is optional. Many protocols will not have a useful inode.
- Moves/renames should be repairable by size, modified time, duration, and later audio fingerprint.
- UI code must not persist or compare raw `smb://` strings.

## Prototypes before UI polish

Run these before building polished screens beyond a basic debug UI:

1. SMB capability spike
   - Add a source by host/share/username/password.
   - List shares and directories.
   - `stat` files.
   - Read byte ranges from a large audio file.
   - Download a file to app storage.
   - Test Synology/QNAP/Windows/macOS if hardware is available.

2. Large scan spike
   - Traverse 10k files with cancellation.
   - Persist checkpoints.
   - Resume after process kill.
   - Keep memory bounded.
   - Insert/update SQLite rows fast enough that the UI stays responsive.

3. Playback cache spike
   - Download one remote audio file to cache.
   - Play with AVPlayer.
   - Continue from the lock screen.
   - Pause/resume from Control Center.
   - Survive route changes and interruptions.

4. Range streaming spike
   - Implement both a minimal `AVAssetResourceLoader` path and a minimal local HTTP range proxy if time allows.
   - Test start latency, seeking, cancellation, content length, unknown length, and cache reuse.
   - Pick one implementation before any UI promises "stream instantly".

5. GRDB/FTS spike
   - Insert 50k tracks and folders.
   - Query path/title/artist/album FTS.
   - Validate migration rollback from a failed migration in development.

No album art grid, custom player chrome, or playlist editor should be prioritized until these pass.

## First vertical slice

The first real slice is:

```text
Fresh install -> add SMB source -> choose root folder -> path scan -> browse Folders -> tap Play Folder -> cache first playable file -> play with AVQueuePlayer -> persist queue -> relaunch and resume queue context
```

Acceptance criteria:

- Fresh install can reach the first playable track in under 2 minutes on a normal local SMB share.
- Source credentials are in Keychain only.
- Local Network permission copy is present and human-readable.
- Folder rows show enough state to distinguish unscanned, scanning, playable, cached, and offline-missing.
- Current-folder playback works before recursive scan is complete.
- Queue state survives app restart.
- Cached audio continues from the lock screen with working play/pause.
- Logs and diagnostics do not expose host passwords, usernames in URLs, or credential tokens.

Explicitly exclude from this slice:

- album/artist screens
- M3U import/export
- durable offline-pack UI
- WebDAV/FTP/SFTP
- video playback
- VLCKit
- custom equalizer, lyrics, CarPlay, Chromecast

## Ordered build plan

1. App shell and tooling
   - Add XcodeGen `project.yml`.
   - Create the SwiftUI app target.
   - Create the local package and empty module targets.
   - Add app entitlements and Info.plist purpose strings for local network and background audio.
   - Add CI/local scripts for build, test, format, and generated-project validation.

2. Domain and store foundation
   - Define identity, source, folder, media, queue, cache, and scan state types.
   - Add GRDB.
   - Create initial schema migrations.
   - Add repositories for sources, folders, media items, queue entries, cache records, and scan checkpoints.
   - Add test fixtures and an in-memory store path.

3. Fake filesystem and debug library
   - Implement an in-memory `RemoteFileSystemClient`.
   - Add fixture directories under `Tools/fixtures`.
   - Build a minimal Folders screen against fake data.
   - This keeps UI agents unblocked without requiring NAS access.

4. SMB source setup
   - Integrate `SMBClient` in `SMBRemote`.
   - Implement Keychain-backed credentials in `BetterStreamingSources`.
   - Add manual source add/test/list-root flow.
   - Add source health result mapping.

5. Path-first scanner
   - Implement recursive scan into `MediaStore`.
   - Add progress, cancellation, checkpoint resume, and subtree scan.
   - Add natural sort keys.
   - Index filename and path into FTS.

6. Folders product surface
   - Replace fake Folders data with store-backed data.
   - Add folder actions: Play, Shuffle, Play Recursively, Shuffle Recursively.
   - Recursive actions may fill the queue progressively; playback starts when the first playable item is ready.

7. Cache-first playback
   - Implement file cache records and app-storage paths.
   - Download current track to cache.
   - Play cached audio through `AVQueuePlayer`.
   - Set file protection appropriate for lock-screen playback.
   - Add audio session, Now Playing, remote commands, interruptions, and route handling.

8. Queue persistence and controls
   - Implement play next, append, reorder, clear.
   - Add shuffle/repeat all/repeat one.
   - Persist queue and current item context.
   - Restore queue after app restart.

9. Range streaming decision
   - Finish resource-loader vs local-proxy prototype.
   - Pick one and implement it in `StreamBridge`.
   - Add prefetch for current track plus next 1-3 queue items.
   - Keep cache-first fallback for every stream failure.

10. Offline cache UX
   - Pin files, folders, recursive folders, and playlists.
   - Add Downloads tab with active queue, failures, storage budget, and retry.
   - Add Offline Mode filter.
   - Add quota and eviction policy.

11. Metadata and search
   - Add throttled tag extraction.
   - Add embedded and folder artwork.
   - Backfill artist/album/title FTS.
   - Add Songs, Albums, Artists, and global Search.

12. Playlists
   - Add app playlists that can contain files and folder references.
   - Add live folder playlists.
   - Add M3U import/export.
   - Connect playlist offline pinning to `CacheManager`.

13. WebDAV
   - Add `WebDAVRemote` behind `RemoteFileSystemClient`.
   - Use `URLSession` background downloads where possible.
   - Reuse the same source, scan, queue, cache, and playback flows.

14. Video
   - Add video item classification and Video library surface.
   - Use AVPlayer first.
   - Add PiP and basic subtitle handling only after playback is stable.
   - Add VLCKit compatibility backend only after license/package review and after unsupported-codec cases are documented.

15. Repair and diagnostics
   - Add source/folder repair flow.
   - Add connection speed test.
   - Add redacted debug bundle export.
   - Add user-facing failure categories: server asleep, password failed, local network blocked, file moved, unsupported codec.

## Later parallel workstreams

Start these only after the first vertical slice has merged and the module interfaces are stable.

| Workstream | Owns paths | Can run after | Must not touch |
| --- | --- | --- | --- |
| WebDAV source | `RemoteFileSystem`, `WebDAVRemote`, source setup UI additions | `RemoteFileSystemClient` contract is stable | queue policy, SMB adapter internals |
| Metadata/artwork | `MetadataCore`, media migrations, album/artist UI | path scanner and store schema exist | SMB source setup, playback queue controls |
| Offline packs | `CacheManager`, Downloads UI, quota settings | cache-first playback works | source credential storage, metadata extraction |
| Playlists/M3U | `PlaylistCore`, Playlists UI, playlist migrations | queue item identity is stable | player backend implementation |
| Range streaming | `StreamBridge`, cache byte layer, playback resolver | range prototype decision is made | source setup UI, metadata UI |
| Video playback | video UI, `PlaybackCore` renderer abstraction, optional `VLCKitBridge` | audio queue and cache are stable | SMB credentials, GRDB base migrations except video fields |
| Diagnostics | `Diagnostics`, source health UI, export redaction tests | source setup and RFS errors exist | playback renderer internals except error inputs |
| Performance harness | `Tools/scripts`, fixtures, large-library tests | MediaStore and scanner exist | product UI layout |

Parallel agents should avoid shared migrations unless the migration owner coordinates the sequence. If two workstreams need schema changes, one agent owns the migration numbers and the other waits for the merged schema.

## Testing strategy

### Unit tests

- Domain identity normalization and equality.
- Natural sort and folder flattening order.
- Queue operations and shuffle/repeat determinism with seeded randomness.
- Cache quota, eviction, stale records, and partial-file recovery.
- Error redaction.

### Store tests

- Every migration runs on an empty store and on seeded prior versions.
- FTS updates when paths and metadata change.
- Queue restore works after simulated process kill.
- Scan checkpoints resume without duplicate rows.
- Repair matching does not relink unrelated files with same names.

### Filesystem tests

- `RemoteFileSystem` contract tests run against the fake client and every real adapter.
- Real SMB integration tests are opt-in and read credentials from local environment variables or an untracked config file.
- Test large directories, Unicode names, case collisions, missing files, permission denial, and slow reads.

### Playback tests

- Keep pure queue logic in unit tests.
- Use app-level integration tests for AVFoundation wiring where simulator support is enough.
- Maintain a required manual-device checklist for Local Network permission, lock-screen audio, route changes, interruptions, and background behavior.

### UI smoke tests

- Fresh install.
- Add source with fake or local fixture source.
- Start scan.
- Open Folders.
- Play a folder.
- Relaunch and verify queue context.
- Toggle Offline Mode after caching one track.

Do not block all development on exhaustive UI automation. The high-risk code is source access, scan persistence, cache correctness, and playback state.

Use `os.Logger` for runtime logs. Add one redaction helper and use it everywhere logs or diagnostics touch URLs, usernames, hosts, paths, or credentials.

## Dependency choices

| Need | Decision | Notes |
| --- | --- | --- |
| SQLite | GRDB | Required for FTS, migrations, explicit schema, performance. |
| SMB | SMBClient first | MIT and pure Swift. Keep adapter isolated. |
| SMB fallback | AMSMB2 spike only | Use only if SMBClient fails real NAS tests; settle LGPL dynamic-link requirements first. |
| Playback | AVFoundation first | Best system integration, background audio, AirPlay, power use. |
| Unsupported codecs | VLCKit later | Add behind renderer abstraction after legal/package review. |
| Keychain | Security.framework wrapper | Avoid dependency for a narrow critical API. |
| DI | Manual protocols and environment structs | No DI framework in MVP. |
| Images/artwork | First-party cache | Avoid image pipeline dependency until artwork requirements prove it is needed. |
| Logging | `os.Logger` | Redaction helper is mandatory. |
| Formatting | SwiftFormat | Keep style mechanical and low-conflict. |

Pin SwiftPM dependencies with conservative version ranges. Do not vendor recon repositories into the app.

## iOS target strategy

- Deployment target: iOS 17.0 for MVP.
- Language mode: Swift 6 for first-party modules where dependencies allow it. Isolate non-concurrency-clean dependencies behind adapter modules using narrow `@preconcurrency` imports if needed.
- Devices: iPhone first. iPad must be usable through adaptive navigation, but no separate iPad-first design in MVP.
- Background modes: audio playback is required. Background downloads are only reliable for URLSession-backed protocols such as HTTP/WebDAV; be honest for SMB.
- File protection: cached media needed for active playback should use a protection class that keeps lock-screen playback working after first unlock.
- Permissions: include clear Local Network purpose copy from the first source setup build. Add Bonjour service declarations for SMB discovery when discovery lands.
- Simulator is not enough. Real-device testing is mandatory for Local Network permission, Keychain behavior, lock-screen controls, audio session interruptions, and app suspension.

Do not lower the target to iOS 16 until the first beta slice is complete and the actual API cost is known. No iOS 15 support.

## Builder rules

- Keep every change runnable. A module can be incomplete, but the app should still build on main.
- Add or update tests in the module you touch.
- Keep UI out of service modules and service dependencies out of domain types.
- Do not edit generated Xcode project files by hand.
- Do not introduce a new third-party dependency without adding it to this document or a follow-up dependency decision doc.
- Do not persist raw remote URLs as identity.
- Do not log credentials, full credential-bearing URLs, or unredacted diagnostic bundles.
- If a feature needs a schema change, own the migration and the downgrade/development-reset story.
